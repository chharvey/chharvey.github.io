General Principles. As Nicolas Gallagher states, all the code in a codebase should look like a single person typed it, no matter how many people contribute. There should be an agreed-upon style, which should be striclty enforced (reverting to existing common patterns when unspecified or in doubt). This is the purpose of this coding style guide. Remember that this guide applies to hand-typed code only. Code that is automatically generated by scripts or other means obviously may not abide by the rules set fourth in this guide.

Note that the coding styles prescribed in this styleguide are specific, opinionated styles for this site’s codebase. This is not the rulebook for all projects. However, you may extend this guide to other projects if you wish.

Textual Structure

Whitespace

  • Use whitespace consistently and to improve code readability.
  • Always use spaces, never tabs.
  • Never leave trailing whitespace at the end of a line.
  • Use spaces to horizontally align repeated text.

Line Breaks & Line Length

  • Blank lines are permitted, but only if they do not contain any spaces.
  • Use blank lines only to separate code structures and improve readability.
  • Keep lines of all code between 75–90 characters long to maximize readability.
  • When writing long passages of prose (in HTML, etc.), use hard line breaks to wrap text. This will help diff tools and reduce merge conflicts.

Exceptions

  • Line breaks may be omitted when multiple, parallel code structures can fit on one line.
  • Line breaks may be introduced to improve readability and produce more useful diffs.

Indentation

  • Use exactly two (2) spaces for each indent level.

Code Formatting

HTML Specifics

  • Always use lowercase for names of tags and attributes. Only use capital letters inside attribute values and in text content.
  • Omit spaces immediately inside tags and inside attribute values.
  • Always include a slash, but not a space, for a self-closing tag.
  • Always use double-quotes for attribute values—even one-word values and empty values.
  • Always use single-quotes for strings within attribute values.
  • Close all tags, even if not required.
  • Always use the empty string for boolean attributes.

CSS/Less Specifics

  • Always use lowercase for tag selectors, and for property names and values. Only use capital letters inside strings and for some class selectors.∗
  • ∗Casing for classnames/mixins are discussed elsewhere.
  • Always use under_score_lower_case, never hyphens, for Less variables.
  • Lower case and shorthand hex color values.
  • Always put spaces immediatlely inside braces if they are on the same line.
  • Always put spaces following colons in CSS rules, and following semicolons when multiple CSS rules share a line and in inline styles.
  • Omit spaces between comma-separated property values (to avoid confusion between these and space-separated property values).
  • Always use single-quotes for all strings in CSS.
  • Always use a semicolon to finish a CSS rule, even the last one, even in inline styles.
  • Use the 1TBS brace style:

JS Specifics

  • In scripting (JS, PHP):
    • class names use PascalCase
    • function/method names use camelCase
    • variable names use under_score_lower_case
    • constant names use UNDER_SCORE_UPPER_CASE
  • Always put spaces immediatlely inside braces if they are on the same line.
  • Always put spaces before open parentheses and after control keywords, but never after names of functions.
  • Always use single-quotes for all strings in JS.
  • Use the 1TBS brace style and npm’s coding style. (In case of conflict, this styleguide overrides npm’s style.)

Accessibility

  • Always provide the html[lang] attribute.
  • Always provide text for the img[alt] attribute.
  • Provide text for the a[title] attribute if the link has non-textual content.
  • For table elements:
    • insert the caption, thead, and tfoot elements (in that order) before the tbody element(s)
    • always nest tr elements inside either thead, tfoot, or tbody elements; never directly inside table
    • always nest th and td elements directly inside tr elements
    • only use th elements when there is a single row and single column of headers (the single row must be in exactly one of the thead, tfoot, or tbody elements)
    • always provide the th[scope] attribute

  • When adding a link to one element, wrap the <a> tag around the outside of the element to be linked.

    There are a few exceptions:

    • when inside the element there is more content not to be linked
    • when the link is marked up with microdata and needs to be nested inside the correct item
  • Use exactly one dfn element per vocabulary term per page, and always provide the dfn[id] attribute.
  • Use exactly one abbr element per abbreviation/acronym per page, and always provide the abbr[id] and abbr[title] attributes.

Optimization

  • Never use any of the following elements or attributes. Use CSS to achieve the desired effect.
    • font
    • hr
    • [align]
    • [border]
  • Omit the [type] attribute from the <link>, <style>, and <script> tags, but only if the value is default (type="text/css" or type="text/javascript")
  • Attributes shall have the following precedence and thus shall be written in the following order:
    1. class
    2. id
    3. any attributes starting with data-
    4. any other global attributes not on this list (style, title, lang, etc.) in any order
    5. all element-specific attributes (href, src, rel, alt, etc.) in any order
    6. microdata attributes in the following order:
      1. itemprop
      2. itemscope
      3. itemtype
      4. itemid
      5. itemref
      Microdata attributes, though global, are listed last to improve readability.

  • Always use unitless line height.

Commenting

Formatting

Comment formatting is best shown by example:

Usage

This section describes the style we will be using to place comments in our codebase. It pertains mostly to CSS code, but it may be used for JavaScript as well. There is not much use for comments in HTML (which, as a markup language, is self-descriptive).

In general, well-commented code has one main function: to provide communication among developers. This means to other people on your team, but also to yourself in the future. Use comments to thoroughly document your code, but from a high level—there is no need to describe every little line or detail. Good code should speak for itself.

Aside: some tools attempt to automatically generate documentation from a rigorous syntax in comments. These are worth looking into.

There are three levels of commenting: those that deliniate the architecture of a file, those that describe individual code structures, and small developer notes.

Architectural

Use comments to create a table of contents and headers in your source code. This is useful for big files that need to be split up into sections. It’s good practice to include the filenames at the top of each partial so that they outline the files when concatenated. See CSS anatomy below for examples.

Structural

Each code structure (pattern, object, component, class, function, etc.) deserves its own comment explaining what it is, how it works, when/why it should be used, and its benefits and limitations. In this comment it’s good practice to also include meta-data such as author and date created/modified.

Developmental

These are small, one-line comments explaining why some particular thing was done. These comments are very likely to be mixed in with the code itself. Use them sparingly, for cases in which the code is not obvious; too many of these comments can be overwhelming. For example, use them when you use !important in CSS, or when you use magic numbers—numbers that just so happen to work in their given situation (like, what does this 18px come from?).

File Anatomy

CSS Stylesheets

Every main stylesheet should have the following components, in the order listed, at the top of the stylesheet.

  1. repository metadata
  2. any direct CSS imports
  3. filename
  4. any (reference) or (inline) Less imports
  5. table of contents
  6. Furthermore, any stylesheet may have any number of section or subsection headings.

Repository Metadata

Include a comment block of information about the repository in the repository’s main file. conf is the example name of the repository.

Filename

Put the filename in a comment block after the meta block and any CSS imports. The filename for the main stylesheet has a slightly different style than that for the partials.

Notice how like the meta block, these comments use block syntax so they get compiled into the output. Lower-level comments not necessary to include in the output use inline comment syntax.

Less Imports

You can use Less imports, reference or inline, to import partials into a main stylesheet. These go after the filename comment block, because any code injected inline should be done so underneath.

Table of Contents

There are two types of table-of-contents. The first type is used in the main stylesheet to describe each imported partial. The second may be used within a partial stylesheet to delineate its sections and subsections.

A table of contents really helps keep your codebase organized and proides a catalog of all the pieces of a project or file. While upkeep maintenance takes some time, this can be seen as an investment for a lot of time and confusion saved later on.

Within partials or sub-files, keep sections and subsections only two levels deep. Keep in mind the Inception rule—three levels is too deep. If you need that kind of complexity, it’s better to split up the stylesheet into smaller files.

Section and Subsection Headings

Use inline comments to begin and end sections and subsections with headings. The hashes (#) make find(-and-replace) much easier. Only subsections and their code are indented.
